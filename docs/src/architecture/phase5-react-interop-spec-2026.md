# Phase 5: Bidirectional Vox↔React Interop Spec (2026)

Parent plan: [External Frontend Interop Plan (2026)](external-frontend-interop-plan-2026.md)

## Goal

Make Vox components and React components first-class citizens of each other's worlds, without a runtime bridge, marshaling layer, or island hydration hack:

1. Vox source can `import react` a `.tsx` component and render it in JSX.
2. Emitted Vox components ship as a standalone npm package consumable by any React app.
3. `@island` is retired; `vox migrate drop-island` rewrites existing usages.

---

## 1. Vox Imports React — Syntax

### 1.1 Proposed syntax

```vox
// vox:skip
import react MyButton from "../ui/MyButton.tsx"
import react { Card, CardHeader } from "../ui/shadcn/card.tsx"
```

The `react` keyword after `import` is the discriminator. It signals to the compiler that:
- The path is a `.tsx` / `.ts` / `.d.ts` file, not a Vox module.
- The imported binding is a React component or hook, not a Vox function.
- No Vox type-checking is applied to the component's internal body.

**Why this over alternatives:**
- `import tsx MyButton from "..."` — too file-format-specific; the import could come from a `.ts` file.
- `import external react MyButton from "..."` — too verbose; `react` already implies external.
- A bare `import MyButton from "../ui/MyButton.tsx"` with `.tsx` extension detection — ambiguous: Vox modules could in principle be emitted as `.tsx`. An explicit keyword is unambiguous and greppable.

Usage in a component body is normal JSX:

```vox
// vox:skip
import react MyButton from "../ui/MyButton.tsx"

component SaveBar(on_save: () -> Unit) {
    view: <div class="bar">
        <MyButton label="Save" onClick=on_save />
    </div>
}
```

### 1.2 Props type bridging — chosen approach

**Option A:** compiler reads `.tsx` / `.d.ts` via a bundled TS type parser.
**Option B:** `vox import-types <path>` materializes Vox type aliases in a sidecar `.vox` file.
**Option C:** types are structurally inferred at emit time with no upfront resolution.

**Decision: Option B.**

Rationale:
- Option A requires shipping a full TS type parser inside the Vox compiler (large dependency surface, complex to keep in sync with TS versions).
- Option C provides no editor assistance and defers all type errors to tsc; defeats the purpose of Vox's type system for prop-wiring bugs.
- Option B produces a human-readable sidecar that can be committed, diffed, and reviewed. The sidecar is regenerated on demand — it is not hand-edited (matches the pattern of `vox emit client`).

**Sidecar workflow:**

```
vox import-types ../ui/MyButton.tsx
# writes: ../ui/MyButton.vox.types (or vox-types/MyButton.vox)
```

The sidecar contains Vox type aliases for the component's props:

```vox
// vox:skip — generated by `vox import-types`; do not edit by hand
type MyButtonProps = { label: str, onClick: () -> Unit, disabled: Option[bool] }
```

The `import react` declaration in Vox source picks up these types automatically when the sidecar exists alongside the source. When no sidecar exists, the props type is `any` and the compiler emits a warning.

**CI recommendation:** run `vox import-types --check` (exits non-zero if sidecars are stale) in the same job that runs `vox emit components --check`.

### 1.3 Emitted TS output

The `import react` declaration passes through as a verbatim React import in the emitted `.tsx`. No runtime marshaling, adapter objects, or wrapper components are injected:

```ts
// emitted SaveBar.tsx
import React from "react";
import MyButton from "../ui/MyButton";  // .tsx stripped per bundler convention

export interface SaveBarProps {
  on_save: () => void;
}

export function SaveBar({ on_save }: SaveBarProps): React.ReactElement {
  return (
    <div className="bar">
      <MyButton label="Save" onClick={on_save} />
    </div>
  );
}
```

The compiler's import resolution pass (§5.2) maps `import react MyButton from "../ui/MyButton.tsx"` → a standard ES import. No `vox-runtime` import is emitted.

## 2. Emitted Vox Components as React Packages

### 2.1 Output directory layout

`vox emit components` (new subcommand) writes to `vox-emit/components/`:

```
vox-emit/components/
  package.json          # generated; do not hand-edit
  index.ts              # re-exports all public components
  types.ts              # shared prop types
  BlogHome.tsx
  PostEditor.tsx
  SaveBar.tsx
  ...
```

Each `*.tsx` is a standalone React component file following the same shape as today's `generate_component` output in `codegen_ts/component.rs` (one `export interface *Props`, one `export function *`).

### 2.2 Generated `package.json`

```json
{
  "name": "@my-app/vox-components",
  "version": "0.3.1",
  "type": "module",
  "exports": {
    ".": {
      "import": "./dist/index.js",
      "require": "./dist/index.cjs",
      "types": "./dist/index.d.ts"
    }
  },
  "peerDependencies": { "react": ">=18" },
  "devDependencies": { "typescript": "^5", "vite": "^5" }
}
```

`name` and `version` are read from `Vox.toml [package]`. The package has **zero `vox-runtime` imports** in component code. Components that call `@mutation` / `@query` endpoints receive typed call wrappers sourced from the Phase 1 client SDK (imported as a peer, not bundled).

Example — a button bound to a mutation:

```vox
// vox:skip
import react { useMutation } from "../api-client/hooks.ts"

component PublishButton(title: str) {
    view: <button onClick=() => publish_post(title)>"Publish"</button>
}
```

Emitted `PublishButton.tsx` imports `publishPost` from `@my-app/api-client` (the Phase 1 SDK), not from any Vox internal. The type of `publishPost` is fully resolved against `Post` and `Id[Post]` from the shared types package.

### 2.3 `index.ts` and `types.ts`

`index.ts` re-exports all `@pub` components:

```ts
export { BlogHome } from "./BlogHome";
export { PostEditor } from "./PostEditor";
export { PublishButton } from "./PublishButton";
```

`types.ts` collects shared prop types that appear in more than one component file. The compiler deduplicates across the module before writing.

## 3. Re-emit Stability

### 3.1 Default behavior

`.tsx` files under `vox-emit/components/` are generated artifacts. The `.vox` source is the single source of truth. Generated files must not be hand-edited (same policy as `SUMMARY.md`, `architecture-index.md`).

### 3.2 Escape hatch — user edit blocks

For the narrow case where a developer needs to inject non-expressible logic into an emitted file (e.g., a third-party SDK call with unusual calling convention), a guarded annotation is supported:

```ts
// @vox:user-edit-begin
const analytics = window.__analytics?.track("view");
// @vox:user-edit-end
```

Re-emit preserves the verbatim content between these markers. Rules:
- Only one block per file.
- The block is injected into a deterministic position (after the last `const` hook call, before the `return`).
- Markers are validated on re-emit; mismatched or nested markers are a hard error.

### 3.3 CI check

```
vox emit components --check
```

Behaves identically to `vox-doc-pipeline --check`: emits to a temp directory, diffs against committed state, exits non-zero if any file differs. Designed to run in the same CI job as `cargo test`.

## 4. `@island` Retirement

### 4.1 What `@island` currently does

From `codegen_ts/island_emit.rs` and `hir/nodes/decl.rs`:

- `@island Name { prop: Type }` declares a client-side interactive component whose props are passed from server-rendered HTML via `data-vox-island` mount attributes and `data-prop-*` attributes.
- The runtime hydrator (`islands/src/island-mount.tsx`) reads these attributes from the DOM and mounts the React component at the `<div data-vox-island="Name" />` marker.
- Islands are tracked in `HirModule.islands: Vec<HirIsland>` (field ownership: `AppContract`).
- In JSX emit, a `<DataChart data=[1,2,3] />` reference to an island name routes through `format_island_mount_ast` → a `<div data-vox-island="DataChart" data-prop-data="[1,2,3]" />` placeholder rather than an inline component call.
- The V1 wire format is locked (`ISLAND_MOUNT_FORMAT_VERSION = 1`) and shared across AST/HIR/Web IR emit paths.

### 4.2 Why bidirectional interop makes `@island` redundant

`@island`'s purpose was to let a Vox `.vox` file reference a React component without the compiler needing to understand that component's internals. The data-attribute bridge was the only available mechanism before first-class `import react` existed.

With Phase 5:
- **Vox → React (inward):** `import react MyButton from "..."` replaces `@island MyButton { ... }`. The component renders inline in the JSX tree; no DOM attribute serialization needed.
- **React → Vox (outward):** external React apps import emitted components directly from the npm package. No hydration marker, no mount script.
- The island hydration runtime (`island-mount.tsx`) becomes dead code once all `@island` sites are migrated.

### 4.3 Migration path

```
vox migrate drop-island
```

This codemod:
1. Finds all `@island Name { prop: Type }` declarations in `.vox` files.
2. Generates a corresponding `.tsx` stub file (e.g., `src/islands/Name.tsx`) with the `*Props` interface and a `TODO: implement` body, if one does not already exist.
3. Rewrites the `@island` declaration to `import react Name from "./src/islands/Name.tsx"`.
4. Rewrites `<Name data_prop=val />` JSX in component views to `<Name dataProp={val} />` (drops the `data-prop-*` encoding).
5. Removes `islands/src/island-mount.tsx` from the project if no remaining `@island` declarations exist.
6. Emits a migration report listing every rewritten site.

The codemod is non-destructive: it creates new files and rewrites `.vox` source, but does not delete the old island stubs until the developer runs `vox migrate drop-island --apply`. A dry-run mode is the default.

### 4.4 Retirement deletion checklist

Once `vox migrate drop-island` is stable and all known usages have been migrated, delete these surfaces. Do **not** delete before that — they are active code on a known decommission timeline.

| Surface | Path | Blocked on |
|---|---|---|
| CLI `vox island` command suite | `crates/vox-cli/src/commands/island/` (entire dir: `generate`, `upgrade`, `list`, `cache`, `build`) | `drop-island` codemod stable |
| Island TS emitter | `crates/vox-compiler/src/codegen_ts/island_emit.rs` | See §5.1 — mark `#[deprecated]` first, remove after codemod stable |
| CLI island templates | `crates/vox-cli/src/templates/islands.rs` | After `vox island` commands removed |
| Island golden example | `examples/golden/v0_shadcn_island.vox` | After `island_emit.rs` removed |
| Islands how-to guide | `docs/src/how-to/how-to-islands-and-pages.md` | Replace with migration guide pointing at `vox migrate drop-island` |
| Islands Vite app | `islands/` (repo root) | After all `@island` usages migrated in codebase |

## 5. Implementation Notes

### 5.1 Compiler files touched

| File | Change |
|---|---|
| `crates/vox-compiler/src/hir/nodes/decl.rs` | Add `HirReactImport` (path, local name, sidecar types path); add `react_imports: Vec<HirReactImport>` to `HirModule` with `SemanticCore` ownership. |
| `crates/vox-compiler/src/codegen_ts/component.rs` | `generate_component` / `generate_component_from_web_ir` read `react_imports` from module context; emit verbatim ES `import Name from "path"` lines before the React import. |
| `crates/vox-compiler/src/codegen_ts/mod.rs` | New `emit_components_package` entry point (writes `package.json`, `index.ts`, `types.ts`, per-component files). |
| `crates/vox-compiler/src/codegen_ts/island_emit.rs` | Mark `format_island_mount_ast`, `island_mount_hir_fragment` as `#[deprecated]` once migration tooling ships; remove after `drop-island` codemod is stable. |
| `crates/vox-compiler/src/` (new file) | `import_react_resolver.rs` — compiler pass that resolves `import react` declarations, reads the sidecar if present, and populates `HirReactImport.props_type`. |

### 5.2 New compiler pass — import-react resolution

Runs between parsing and type-checking, analogous to the existing import resolution pass for Vox modules. For each `import react` declaration:

1. Resolve the file path relative to the source file.
2. Look for a sidecar file (`<path>.vox.types` or `vox-types/<basename>.vox`).
3. If found, parse the sidecar's `type *Props = { ... }` alias and attach it to the `HirReactImport` node.
4. If not found, attach `HirType::Any` and emit a `W0401: react import has no type sidecar` warning.
5. During type-checking, JSX attribute types for the imported component are resolved against the attached `HirType`.
